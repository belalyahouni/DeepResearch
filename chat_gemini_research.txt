In the context of your COMP3011 assignment, MCP stands for Model Context Protocol.

It is a relatively new open standard introduced by Anthropic that allows AI models (like Claude, or the GenAI tools you are encouraged to use) to connect directly to external data sources and tools.

Why is it in your assignment brief?

The brief mentions an MCP server as a modern alternative to traditional web hosting (like PythonAnywhere). Since your assignment is "Green Light" for GenAI and rewards "creative, high-level use of AI," implementing an MCP server is a way to hit those higher grade bands (70–100 range).

How it works for your project:

Normally, a Web API works like this:

Traditional: Your API (REST/GraphQL) <--> A Client (Browser/Postman).

MCP: Your API <--> An AI Tool (like Claude Desktop or a custom LLM interface).

If you build an MCP Server, you are essentially making your database "readable" and "actionable" by an AI. For example, if you built the "Productivity and Habit Analytics API" mentioned in the ideas section:

You create an MCP server that exposes your "habit" data.

You connect that server to a GenAI interface.

You can then ask the AI: "Look at my habit data from the last week and tell me which day I was most productive."

The AI uses the MCP to "reach into" your database, grab the data, and analyze it for the user.

Why you might choose to do it:

Grade Boost: The marking criteria specifically list "MCP-compatible" under the 70–79 (Very Good) band for Content.

Cutting Edge: It proves you are engaging with "contemporary technologies" and "emerging standards," which the "Outstanding" (90-100) category looks for.

Functionality: It makes your project a "plugin" for AI, rather than just a standalone website.

A standard API is a "manual tool"—you (the developer) have to write every line of code that says "First call the optimizer, then the search, then the summarizer."

An MCP is a "smart toolbox"—you just put the tools in the box, and the AI decides which one to use and when.

Here is the realistic workflow and why it actually matters for your project.

The Use Case: "The 2:00 AM Literature Review"

Imagine a student or researcher is using Claude Desktop (or a custom LLM interface) with your Research Agent MCP Server installed.

1. The Conversation (The "Point")

Instead of clicking buttons in your custom UI, the user just types:

"Hey, find me the most recent papers on 'Transformer cooling systems' from OpenAlex. Summarize the top 3, but if any of them are just theoretical and don't have experimental data, skip them and find more."

2. The MCP Workflow (Under the Hood)

Because you implemented MCP, your agents are now "Tools" that the AI understands. It doesn't follow a hard-coded script; it reasons through the tools you gave it:

Step 1: The AI sees your prompt_optimizer tool. It uses it to turn "Transformer cooling systems" into a better search query.

Step 2: It calls your openalex_search tool.

Step 3: It looks at the results. It realizes two papers are theoretical.

Step 4: Crucial Step: Instead of crashing or needing a human, the AI decides to call openalex_search again with a filtered query for "experimental data"—because it knows that tool exists in your MCP server.

Step 5: It finally calls your summarize_agent on the correct papers.

Why this is better for your grade (and for life):

As of March 2026, there is no direct "NotebookLM API" that works as a one-to-one replacement for the consumer app. NotebookLM remains a standalone research product.

However, since you are already using the Gemini API and Python, you can replicate 99% of NotebookLM’s functionality using a feature called Context Caching.

Why Context Caching is your "NotebookLM API"

NotebookLM works by loading documents into a "Notebook" so the AI can reference them instantly without forgetting. In a standard API, sending 50 research papers every time you ask a question would be incredibly slow and expensive.

Context Caching solves this by:

Uploading Once: You send your OpenAlex papers/summaries to the Gemini API one time.

Storing the "Notebook": The API caches these tokens (up to 1 million+) on Google’s servers.

Instant Q&A: When your "Chat Agent" asks a question, it only sends the new question. The API "links" it to the cached papers instantly.

How to Implement This in Your Framework

NotebookLM Feature	Your Framework (API Implementation)
The "Notebook"	A CachedContent object in the Gemini API.
Sources	Your OpenAlex search results uploaded as a single text/PDF blob.
Grounded Answers	Setting a system_instruction in the cache: "Answer only using the cached papers."
Cost Efficiency	Caching gives you a 90% discount on input tokens for repeated queries.
Example Implementation (Python)

Instead of a simple chat, your "Chat Agent" would use the caching service:

Python
from google import genai

# 1. Take your OpenAlex summaries and create a 'Notebook'
# This stays on Google's server so you don't re-send it every time
cache = client.caches.create(
    model="models/gemini-1.5-flash",
    config=types.CreateCachedContentConfig(
        display_name="research_notebook_cooling_systems",
        system_instruction="You are a research assistant. Use the provided OpenAlex papers to answer.",
        contents=[openalex_papers_blob],
        ttl_seconds=3600  # Keep the 'notebook' alive for 1 hour
    )
)

# 2. Your Chat Agent now 'talks' to that specific cache
response = client.models.generate_content(
    model="models/gemini-1.5-flash",
    contents="Based on these papers, what is the consensus on immersion cooling?",
    config=types.GenerateContentConfig(cached_content=cache.name)
)
