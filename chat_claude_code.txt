‚Źļ The core issue is that your current model only stores a total interaction_count ‚ÄĒ
  there's no per-event history to filter by time window. Here are two approaches:

  ---
  Option A: Interaction Log Table (recommended)

  Add a new CommunityInteraction table ‚ÄĒ one row per interaction event:

  class CommunityInteraction(Base):
      __tablename__ = "community_interactions"
      id: Mapped[int] = mapped_column(primary_key=True, autoincrement=True)
      arxiv_id: Mapped[str] = mapped_column(String, index=True, nullable=False)
      interacted_at: Mapped[datetime] = mapped_column(DateTime,
  server_default=func.now(), index=True)

  Then GET /community gains a period query param (week / month / year / all), and the
  query becomes:

  cutoff = datetime.utcnow() - timedelta(days=7)  # or 30, 365, None

  stmt = (
      select(CommunityInteraction.arxiv_id, func.count().label("count"))
      .where(CommunityInteraction.interacted_at >= cutoff)
      .group_by(CommunityInteraction.arxiv_id)
      .order_by(desc("count"))
      .limit(limit)
  )

  CommunityPaper stays as-is for all-time totals and last_interacted_at. The
  interaction_count column there just gets incremented as it does now.

  Pros: Accurate, flexible, no background jobs, can add more periods later.
  Cons: The table grows unboundedly ‚ÄĒ you'd eventually want to prune rows older than a
  year.

  ---
  Option B: Denormalised counters on CommunityPaper

  Add three extra columns to the existing model:

  interactions_week: Mapped[int] = mapped_column(Integer, default=0)
  interactions_month: Mapped[int] = mapped_column(Integer, default=0)
  interactions_year: Mapped[int] = mapped_column(Integer, default=0)

  Increment all four on each interaction, then reset the shorter windows on a
  cron/background task.

  Pros: Fast reads, no join/aggregation needed.
  Cons: Needs a reset job (which resets at a fixed calendar boundary, not a rolling
  window), and counts are approximate after reset.

  ---
  Recommendation

  Go with Option A. It's a rolling window (not "since last Monday"), which is far more
  useful for ranking. The table stays small if you prune anything older than a year.
  The period query param is a clean API addition:

  GET /community?period=week
  GET /community?period=month
  GET /community?period=year
  GET /community        # defaults to all-time (existing behaviour)

  Want me to implement Option A?
